You are building the SimplyDSE Communication system.

IMPORTANT:
Do NOT build a full Slack/WhatsApp-style chat system yet.
Do NOT build unnecessary real-time complexity at this stage.
Do NOT create casual social messaging.

Build a focused enterprise “Compliance Inbox” system designed for workplace DSE assessment communication, HR follow-ups, training reminders, and risk-related conversations.

This is a communication module for a multi-tenant workplace compliance SaaS platform.

==================================================
PRIMARY GOAL
==================================================

Build a professional internal communication interface for SimplyDSE that allows employees, HR users, managers, and compliance teams to communicate securely inside their organisation.

The system should support:

- assessment-related conversations
- HR follow-ups
- employee support messages
- risk review discussions
- training reminders
- department-level communication
- compliance action threads
- basic announcements

This should feel like:

“a secure enterprise compliance inbox”

NOT:

“a casual chat app”.

==================================================
CURRENT PROJECT CONTEXT
==================================================

SimplyDSE is a multi-tenant enterprise SaaS platform for:

- DSE assessments
- workplace compliance
- employee wellbeing
- ergonomic risk tracking
- HR follow-ups
- reporting
- training
- audit records

The messaging system must support this compliance workflow.

==================================================
IMPORTANT ARCHITECTURE RULE
==================================================

Every message, conversation, thread, notification, and attachment must belong to an organisation.

Every communication record must include:

- organisation_id
- sender_id
- recipient/participant data
- timestamps

Do NOT allow cross-tenant messaging.

Users from one organisation must never see messages from another organisation.

==================================================

==================================================
ROLE-BASED COMMUNICATION LOGIC
==================================================

Employee users can:

- view conversations they are part of
- reply to HR or manager messages
- view messages linked to their assessments
- receive training reminders
- receive announcements
- send support/request messages to HR

Employees must NOT:

- view organisation-wide conversations
- message all staff
- access HR-only discussions
- see conversations between HR and other employees

==================================================

HR users can:

- message employees
- create assessment follow-up threads
- create training reminder messages
- view organisation-level HR communication
- communicate with managers
- send announcements if permitted
- view conversations related to employees in their organisation

==================================================

Managers can:

- message HR
- view team-related communication only
- message employees assigned to their team
- participate in department-level threads

==================================================

Organisation Admins can:

- manage organisation-level communication
- view compliance-related conversations within the organisation
- send announcements
- manage communication settings

==================================================

Super Admin users should NOT casually read tenant messages.

If support access is needed later, it must be implemented as a controlled audited support mode.

Do not expose tenant messages to Super Admin by default.

==================================================
SUPABASE RLS REQUIREMENTS
==================================================

Implement or preserve RLS policies.

Users can only access conversations where:

- conversation.organisation_id matches their organisation_id
AND
- they are listed in conversation_participants

HR/Admin users can access organisation-level compliance conversations according to role permissions.

Employees can only access their own conversations.

No cross-tenant message access is allowed.

==================================================
MESSAGE CREATION LOGIC
==================================================

When a user sends a message:

1. Validate authenticated user
2. Validate organisation_id
3. Validate conversation access
4. Insert message
5. Update conversation last_message_at
6. Create notification records for other participants
7. Update unread state

Do not allow users to send messages into conversations they are not part of.

==================================================
ASSESSMENT-LINKED CONVERSATIONS
==================================================

When an assessment requires HR review, the system should be able to create an assessment follow-up conversation.

Example:

Employee completes Hybrid DSE Assessment
↓
Risk level = High
↓
System creates thread:
“Follow-up required for Hybrid DSE Assessment”
↓
Participants:
- employee
- HR user
- manager where applicable

The conversation should link to:
- assessment submission
- employee profile
- risk result
- report PDF
